Entendiendo numba: el modo nopython

Como podemos leer en la documentaci√≥n, numba tiene dos modos de funcionamiento b√°sicos: el modo object y el modo nopython.

  • El modo object genera c√≥digo que gestiona todas las variables como objetos de Python y utiliza la API C de Python para operar con ellas. En general en este modo no habr√° ganancias de rendimiento (e incluso puede ir m√°s lento), con lo cual mi recomendaci√≥n personal es directamente no utilizarlo. Hay casos en los que numba puede detectar los bucles y optimizarlos individualmente (loop-jitting), pero no le voy a prestar mucha atenci√≥n a esto.
  • El modo nopython genera c√≥digo independiente de la API C de Python. Esto tiene la desventaja de que no podemos usar todas las caracter√≠sticas del lenguaje, pero tiene un efecto significativo en el rendimiento. Otra de las restricciones es que no se puede reservar memoria para objetos nuevos.

Por defecto numba usa el modo nopython siempre que puede, y si no pasa a modo object. Nosotros vamos a forzar el modo nopython (o ¬´modo estricto¬ª como me gusta llamarlo) porque es la √∫nica forma de aprovechar el potencial de numba.

√Åmbito de aplicaci√≥n

El problema del modo nopython es que los mensajes de error son totalmente inservibles en la mayor√≠a de los casos, as√≠ que antes de lanzarnos a compilar funciones con numba conviene hacer un repaso de qu√© no podemos hacer para anticipar la mejor forma de programar nuestro c√≥digo. Pod√©is consultar en la documentaci√≥n el subconjunto de Python soportado por numba en modo nopython, y ya os aviso que, al menos de momento, no tenemos list comprehensions, generadores ni algunas cosas m√°s. "With a few annotations, array-oriented and math-heavy Python code can be just-in-time compiled to native machine instructions, similar in performance to C, C++ and Fortran". [√ânfasis m√≠o]
